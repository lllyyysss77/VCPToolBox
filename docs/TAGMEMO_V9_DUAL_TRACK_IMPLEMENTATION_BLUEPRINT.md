# TagMemo V9 双轨工程施工图

> 状态：施工基线  
> 日期：2026-07-14  
> 适用范围：TagMemo V8.3 → V9 并轨、派生资产固化、LightMemo 寻址 A/B  
> 核心原则：结果优先、生产冻结、低成本、低功耗、可回退、可证伪

---

## 1. 北极星目标

VCP 推进 TagMemo 的目的不是构造一套形式完备但脱离使用结果的理论，也不是以论文指标代替真实记忆体验。

唯一核心目标是：

> 以尽可能低的计算成本、功耗和系统复杂度，让 AI 在被动检索过程中表现出接近人的直觉回忆、逻辑联想、跨概念桥接和语境补全能力。

因此，本工程以实际记忆寻址结果作为最高裁决依据。

理论、测量和数学结构的价值在于：

1. 帮助发现有效机制；
2. 约束系统不要失控；
3. 帮助复现和迁移收益；
4. 帮助识别系统何时不应出手；
5. 降低后续维护与调参成本。

它们不能替代真实结果，也不能成为无限增加工程复杂度的理由。

---

## 2. 已确认的工程判断

### 2.1 V8.3 的地位

当前 V8.3 是经过实际用户、企业知识库和科研数据库反馈验证的生产基线。

对它的工程定性是：

- 相比直接 KNN，在绝大多数使用情境中具有正收益；
- 已形成一组尚未完全归因、但确实有效的机制组合；
- 当前参数与特定 embedding 模型和真实数据分布存在经验拟合关系；
- 不能为了统一数学表达而一次性改写生产行为。

所以 V8.3 不是待清理的旧代码，而是 V9 的对照组和安全回退路径。

### 2.2 Potential Field 的地位

当前所谓“测地线重排”实际是查询条件化标签能量场对候选文档的响应，不是真正的图测地距离。

其工程定性是：

- 条件性高收益；
- 高方差；
- 地图可靠时可能产生远超 KNN 的桥接召回；
- 地图不完整或伪边较多时可能产生明显负收益；
- 必须保留硬防御和拒判能力。

V9 不以“让它更频繁出手”为目标，而以“让它在正确的时候出手”为目标。

### 2.3 V9 的存在条件

V9 不能仅因数学表达更统一就获得生产资格。

V9 必须先证明：

1. 与 V8.3 在相同输入上确实产生可观察差异；
2. 差异发生在真实记忆寻址结果，而非只有内部数值变化；
3. 差异能被人类或 Agent 主观判断为收益；
4. 收益能够在多个查询或数据域中重复出现；
5. 计算成本、延迟和内存增长与收益相称；
6. 失败时可以无损回退到 V8.3。

## 3. 总体施工策略

工程采用：

> 单一事实底座 + 双版本派生资产 + 双版本纯查询算子 + 单生产出口 + LightMemo 同条件 A/B。

### 3.1 单一事实底座

V8.3 和 V9 共享以下不可分叉的原始事实：

- 标签 ID 与标签文本；
- 标签 embedding；
- 文件—标签关系；
- 标签叙事顺序；
- chunk embedding；
- 文件和 chunk 元数据；
- embedding 模型签名；
- 原始共现事件。

禁止为 V9 复制一套独立原始知识库。

### 3.2 双版本派生资产

允许按版本分别生成：

- V8.3 兼容残差增益；
- V9 原始残差比例；
- V9 固定映射锚增益；
- V8.3 累计共现矩阵；
- V9 有界传播核；
- 各自的 artifact signature；
- 各自的处理状态和生成代际。

### 3.3 单生产出口

生产查询每次只能选择一个活动版本。

活动版本由 [`rag_params.json`](../rag_params.json) 热配置决定，但每个请求开始后必须固化版本快照，不能在同一请求中被热更新改变。

默认活动版本保持 V8.3，直到 V9 完成验收。

### 3.4 A/B 旁路

LightMemo A/B 必须显式同时调用两个版本，不能通过临时修改全局活动版本实现。

A/B 结果只用于测绘和评估，不得隐式影响生产输出。

## 4. 不可违反的工程纪律

### 4.1 生产冻结

在 V9 获得生产资格前：

- V8.3 的查询增强公式不变；
- V8.3 的传播行为不变；
- V8.3 的重排门控不变；
- V8.3 的默认配置不变；
- P0 数据修复必须提供兼容映射，保证 V8.3 输出行为可对照。

### 4.2 同条件对照

每次 V8.3/V9 A/B 必须共享：

- 同一个原始查询文本；
- 同一个原始 query embedding；
- 同一个搜索范围；
- 同一个 BM25 开关；
- 同一个候选池；
- 同一个 K；
- 同一批 chunk 向量；
- 同一时刻的数据快照；
- 同一个 core tag 输入。

禁止分别重新向量化，禁止分别重新采样候选。

### 4.3 不使用全局切换完成 A/B

禁止采用以下流程：

1. 修改全局版本为 V8.3；
2. 执行一次查询；
3. 修改全局版本为 V9；
4. 再执行一次查询。

这种实现会污染并发请求，并使 A/B 失去一致性。

### 4.4 版本显式传递

版本必须作为调用参数或请求快照显式传递。

建议统一使用：

- `v8_3`
- `v9`

任何未知版本必须拒绝或安全回退到配置中的生产版本，并记录警告。

### 4.5 低成本优先

新增机制必须记录：

- 预计算耗时；
- 查询额外耗时；
- 内存增量；
- SQLite 增量；
- Rust/JS 边界调用次数；
- 是否引入额外 embedding 请求。

默认不允许 V9 为一次普通查询增加新的远程模型调用。

## 5. 目标架构

### 5.1 事实层

事实层由 [`KnowledgeBaseManager.js`](../KnowledgeBaseManager.js) 和 SQLite 管理。

职责：

- 维护原始标签、文件、chunk 和顺序事实；
- 执行兼容 schema 迁移；
- 提供数据库代际信息；
- 管理 Rust 写租约；
- 不包含版本特定传播逻辑。

### 5.2 派生层

派生层由 [`rust-vexus-lite/src/lib.rs`](../rust-vexus-lite/src/lib.rs) 与 [`TagMemoEngine.js`](../TagMemoEngine.js) 协同完成。

职责：

- 计算 pairwise similarity；
- 记录低于阈值的负结果；
- 计算单位化后的原始内生残差比例；
- 生成 V8.3 兼容增益；
- 生成 V9 固定锚增益；
- 生成版本化矩阵或传播核；
- 返回实际生效配置和签名。

### 5.3 查询层

查询层由 [`TagMemoEngine.applyTagBoost()`](../TagMemoEngine.js:202) 演进为版本显式的统一入口。

目标接口语义：

- 未指定版本：使用请求开始时捕获的活动版本；
- 指定 `v8_3`：严格执行兼容路径；
- 指定 `v9`：严格执行 V9 路径；
- 返回结果必须携带版本与 artifact signature；
- 查询过程只读派生资产；
- 两个版本的查询状态完全隔离。

### 5.4 搜索层

[`KnowledgeBaseManager.search()`](../KnowledgeBaseManager.js:901) 负责：

- 捕获版本快照；
- 复用预先计算的增强结果；
- 保证联合索引只执行一次版本算子；
- 将版本信息和场响应元数据传递到结果；
- 保持现有 API 的默认兼容性。

### 5.5 测绘层

[`LightMemoPlugin.handleMapping()`](../Plugin/LightMemo/LightMemo.js:560) 保留现有空间距离测绘。

新增“记忆寻址 A/B”模式，负责：

- 固定候选池；
- 并行执行 V8.3/V9；
- 按 chunk ID 对齐；
- 输出排名、分数、召回集合和 fallback 差异；
- 禁止外部 reranker 干扰首轮判断。

## 6. 派生资产数据契约

### 6.1 Artifact Signature

所有正式派生资产必须绑定：

\[
artifactSig =
H(
modelSig,
graphGeneration,
algorithmVersion,
configHash
)
\]

至少包含：

- embedding 模型签名；
- embedding 维度；
- 图数据代际；
- 算法版本；
- 实际生效参数哈希；
- schema 版本。

### 6.2 处理状态

派生任务不能只记录成功结果，还必须记录处理状态：

- `computed`
- `below_threshold`
- `insufficient_neighbors`
- `missing_vector`
- `failed`

缓存完整性的判断对象是“当前 artifact 下是否已经处理”，不是“是否成功产生正值”。

### 6.3 Pairwise 负缓存

低于相似度阈值的 pair 必须写入轻量状态记录，避免增量任务反复计算。

负缓存必须绑定：

- tag pair；
- model signature；
- artifact signature；
- threshold；
- status；
- computed timestamp。

### 6.4 内生残差双语义

必须严格分离：

- `raw_residual_ratio`：数据库无关的原始局部不可解释比例；
- `v8_3_compat_gain`：维持现有生产语义的兼容值；
- `v9_anchor_gain`：由固定函数从 raw 值映射的 V9 工程增益。

原始测量值不能因全库新增极端标签而重标定。

### 6.5 实际配置回传

Rust 派生任务的返回值必须包含：

- 实际 method；
- max neighbors；
- max basis；
- min neighbors；
- semantic gate 参数；
- position decay；
- min gain；
- algorithm version；
- artifact signature。

日志中显示的配置必须与 Rust 实际执行配置一致。

## 7. V9 第一版数学边界

V9 第一版只实施低风险、高收益且可独立对照的数学变化。

### 7.1 有界传播核

V8.3 保留累计共现边权。

V9 将共现证据强度与传播传导率分离：

\[
C_{ij} = \log(1 + cooc_{ij})
\]

\[
P_{ij}
=
\frac{g(C_{ij},S_{ij},D_{ij},A_{ij})}
{\sum_k g(C_{ik},S_{ik},D_{ik},A_{ik})+\epsilon}
\]

第一版采用行归一化有界核，不立即引入复杂稳态求解器。

### 7.2 残差固定映射

V9 使用单位化向量计算：

\[
r_i =
\frac{\|(I-U_iU_i^\top)v_i\|_2}
{\|v_i\|_2+\epsilon}
\]

并夹逼到 \([0,1]\)。

工程锚增益使用可配置但固定的单调映射，不依赖全库 Min-Max。

### 7.3 保留有限跳传播

V9 第一版继续使用有限跳和现有截断结构，以控制变量。

暂不同时引入：

- PPR；
- heat kernel；
- 能量最小化；
- 真正最短路径；
- GNN；
- 端到端学习。

### 7.4 保留硬防御

Potential Field 继续保留：

- 最小场标签数；
- 正能量质量检查；
- 场熵检查；
- 候选覆盖率；
- 最大场响应；
- 响应展宽；
- 最小标签命中数；
- 低信任回退 KNN。

### 7.5 正式命名

V9 对外正式使用 `potentialFieldRerank` 语义。

原有入口保留 deprecated alias，避免破坏现有调用方。

## 8. 配置设计

在 [`rag_params.json`](../rag_params.json) 的 `KnowledgeBaseManager` 下新增 `tagMemoVersioning`。

建议字段：

- `activeVersion`：默认 `v8_3`；
- `availableVersions`：允许值列表；
- `dualArtifactsEnabled`：是否生成双版本派生资产；
- `abMappingEnabled`：是否开放 LightMemo A/B；
- `fallbackVersion`：固定 `v8_3`；
- `includeArtifactMetadata`：是否在开发结果中显示签名。

新增 `v9` 配置块，至少包含：

- 传播核归一化开关；
- 证据压缩函数；
- anchor gain 参数；
- 有限跳参数；
- Potential Field 参数；
- 安全门控参数。

配置热更新只能影响后续请求，不能改变正在执行的请求。

## 9. LightMemo 记忆寻址 A/B

### 9.1 新模式

建议命令：

- `compare_tagmemo_versions`
- `tagmemo_ab`
- `寻址对照`

现有 `map_distance` 保持不变。

### 9.2 输入

复用 LightMemo 搜索参数：

- query；
- maid；
- folder；
- search_all_knowledge_bases；
- k；
- BM25；
- tag_boost；
- core_tags；
- core_boost_factor；
- 是否启用 Potential Field。

A/B 模式强制禁用外部 reranker。

### 9.3 执行顺序

1. 解析一次搜索范围；
2. 收集一次原始候选；
3. 构造一次 BM25 候选池；
4. 获取一次 query embedding；
5. 用 `v8_3` 生成增强结果；
6. 用 `v9` 生成增强结果；
7. 对同一候选池分别计算向量分数；
8. 分别执行版本对应的 Potential Field；
9. 按 chunk ID 对齐；
10. 输出 Top-K 与差异统计。

### 9.4 输出表格

最小字段：

| 字段 | 说明 |
|---|---|
| Chunk | 同一真实记忆片段 |
| 来源 | 日记本与文件 |
| V8.3 排名 | 基线排名 |
| V9 排名 | 新版本排名 |
| 排名变化 | V9 相对升降 |
| 原始 KNN | 原始向量分数 |
| V8.3 分数 | 基线最终分数 |
| V9 分数 | 新版最终分数 |
| V8.3 场响应 | 基线图贡献 |
| V9 场响应 | 新版图贡献 |
| V8.3 fallback | 是否拒判及原因 |
| V9 fallback | 是否拒判及原因 |

附加摘要：

- Top-K 重合率；
- V8.3 独占召回；
- V9 独占召回；
- 平均绝对排名变化；
- 最大排名跃迁；
- 两边查询耗时；
- 两边 artifact signature；
- 两边实际版本。

### 9.5 主观裁决

首轮裁决标签：

- `v8_3_win`
- `v9_win`
- `equivalent`
- `both_bad`
- `v9_surprise`
- `v9_drift`
- `insufficient_map`

首版可以只输出供人或 Agent 手工判断，不强制立即建设持久化评审数据库。

## 10. 分阶段施工顺序

### Phase P0：派生资产正确性

交付：

1. schema 向后兼容迁移；
2. artifact signature；
3. 处理状态；
4. Pairwise 负缓存；
5. skipped Tag 完整性；
6. JS/Rust 配置单一真相源；
7. 原始残差与工程增益分离；
8. V8.3 兼容值保留。

验收：

- 老数据库可直接启动；
- 不删除原始数据；
- V8.3 默认结果不发生非预期漂移；
- 重复派生任务能正确命中正负缓存；
- 模型或参数变化会生成新 artifact。

### Phase V9-A：双轨运行时

交付：

1. 版本配置；
2. 双版本派生资产加载；
3. 显式版本查询；
4. 请求级版本快照；
5. 并发隔离；
6. V8.3 默认生产出口。

验收：

- 同一进程可同时调用两个版本；
- A/B 不修改全局活动版本；
- 一个版本失败不污染另一个版本；
- V9 失败自动回退 V8.3。

### Phase V9-B：首个 V9 数学核

交付：

1. 有界传播核；
2. 固定残差锚映射；
3. Potential Field 正名；
4. 现有硬防御复用；
5. 版本级耗时统计。

验收：

- V9 每节点向外传播总传导率有界；
- 不增加远程 embedding 调用；
- 不移除有限跳熔断；
- 不放宽生产安全门控。

### Phase V9-C：LightMemo 寻址对照

交付：

1. 新 A/B 命令；
2. 同候选池双评分；
3. Chunk 对齐表；
4. 差异摘要；
5. 工具协议文档。

验收：

- 同一请求 embedding 只生成一次；
- 候选 ID 集完全一致；
- 外部 reranker 不参与；
- 输出能直接支持主观胜负判断。

### Phase V9-D：结果裁决

只有出现可重复收益后，才进入：

- 针对差异查询的微观回放；
- 连续可信度校准；
- 虫洞局部显著性；
- 路径冗余；
- 稳态传播比较；
- 真正测地线探索。

若 V9 没有形成可感知收益，则停止扩展理论，不因沉没成本继续推进。

## 11. 回退策略

### 11.1 配置回退

将活动版本切换为 `v8_3`，后续请求立即使用 V8.3。

### 11.2 数据回退

V9 新表或新列只能附加，不得覆盖 V8.3 所需数据。

### 11.3 代码回退

兼容入口和 V8.3 算子在 V9 验收前不得删除。

### 11.4 派生任务回退

V9 派生任务失败时：

- 保留失败状态；
- 不覆盖上一个健康 artifact；
- 不阻断 V8.3 加载；
- 不触发不可控重算循环。

## 12. 性能预算

首轮目标不是绝对数值，而是建立约束。

### 查询期

- 不新增远程模型调用；
- 双轨只在显式 A/B 时执行；
- 普通生产查询只执行活动版本；
- V9 单次查询延迟应控制在 V8.3 的可接受倍数内；
- 不复制完整 chunk 候选文本和向量。

### 预计算期

- Pairwise 负缓存减少重复计算；
- 双版本共享原始 pairwise cosine；
- 双版本共享原始共现事实；
- 只对任务特定的最终核和映射分叉；
- 继续服从 Rust 写租约和启动冷却期。

### 内存

- 不维护两份标签 embedding；
- 不维护两份 pairwise similarity；
- 允许维护两套稀疏传播 Map；
- 内存画像必须分别报告 V8.3/V9 派生资产规模。

## 13. 测试与验收矩阵

### 兼容性

- 老配置无新字段；

## 14. 停工条件

遇到以下情况必须停工审计，不得继续叠加机制：

1. V8.3 生产结果发生无法解释的系统性漂移；
2. V9 A/B 不能保证同候选池；
3. 版本热切换污染并发请求；
4. artifact signature 无法唯一描述实际配置；
5. 派生任务触发反复全表重算；
6. V9 增加新的远程模型依赖；
7. V9 内存或延迟增长明显高于收益；
8. 为解释无差异结果而不断增加新机制。

## 17. 工程宣言

TagMemo 的价值不在于证明一个公式可以描述记忆，而在于用可承担的成本，让 AI 实际表现出更好的记忆。

我们允许机制先于理论被发现，允许直觉先于归因产生价值，也允许系统在地图不可靠时拒绝出手。

但我们不允许：

- 用漂亮理论掩盖无收益；
- 用内部指标替代真实寻址结果；
- 用复杂度替代进步；
- 用不可回退的重构破坏已经有效的系统。

本轮施工的最终原则是：

> V8.3 不必先被完全解释；V9 必须先证明自己值得被解释。

以及：

> 结果不是理论的附属验收项，结果就是 TagMemo 工程真理的最高来源。

## 18. 施工补丁：原子发布、双模式 A/B 与版本失败语义

> 本节为 2026-07-14 追加的强制施工约束；若与前文存在歧义，以本节为准。

### 18.1 原子不可变 ArtifactBundle

请求不得只捕获版本字符串。每次查询必须持有一个完整、不可变的资产包引用：

```javascript
{
  version,
  artifactSig,
  graphGeneration,
  effectiveConfig,
  residualMap,
  propagationKernel,
  pairwiseView,
  potentialFieldConfig
}
```

派生资产采用类似 RCU 的发布方式：

1. 在 staging 对象中构造完整资产；
2. 校验签名、代际、状态覆盖率、数值范围和矩阵完整性；
3. 一次性替换当前 `ArtifactBundle` 引用；
4. 已开始请求继续持有旧对象；
5. 新请求读取新对象；
6. 不在已发布对象上原地修改 Map、配置或签名。

因此，请求快照的正式定义是：

> 版本 + artifact generation + 不可变对象引用。

禁止把 residual、pairwise、matrix 和配置分别热替换到活动查询视图中。

### 18.2 A/B 模式 A：固定对称超集测核

模式 A 用于隔离两个版本在相同候选集合上的评分差异。

候选超集按以下来源对称构造：

\[
\mathcal C_{\mathrm{AB}}
=
\operatorname{TopL}(q)
\cup
\operatorname{TopL}(q'_{8.3})
\cup
\operatorname{TopL}(q'_9)
\cup
\mathcal C_{\mathrm{BM25}}
\]

约束：

- \(L\) 必须明显大于最终 \(K\)；
- 两个版本必须对同一个并集重新计算分数；
- 每个候选必须记录进入超集的来源路径；
- 候选来源至少区分原始 KNN、V8.3 增强向量、V9 增强向量和 BM25；
- 不得仅使用原始 \(q\) 的候选池，否则会提前裁掉 V9 的召回扩张收益。

模式 A 回答：

> 在双方都拥有公平召回机会、且最终候选集合相同的条件下，哪个核排得更好？

### 18.3 A/B 模式 B：端到端独立寻址测产品

模式 B 分别执行两个版本的真实寻址链：

\[
q'_{8.3}\rightarrow \mathcal C_{8.3}
\]

\[
q'_9\rightarrow \mathcal C_9
\]

输出层再按 chunk ID 对齐并报告：

- 共同召回；
- V8.3 独占召回；
- V9 独占召回；
- 排名变化；
- 独占结果的实际价值；
- 明显主题漂移；
- 每侧实际版本与失败状态。

模式 B 回答：

> 用户真实使用时，V9 是否比 V8.3 找到了更有价值的记忆？

V9 获得生产资格必须通过模式 B；模式 A 不能替代端到端产品验证。

### 18.4 严格版本与生产回退

版本调用分为三类。

#### 生产隐式调用

调用方未指定版本时，使用请求开始时活动 `ArtifactBundle` 的版本。

活动 V9 失败时允许保命式回退到 V8.3，但结果必须携带：

```javascript
{
  requestedVersion: "v9",
  effectiveVersion: "v8_3",
  fallbackApplied: true,
  fallbackReason
}
```

#### A/B 显式调用

显式指定版本并启用严格模式：

```javascript
{
  version: "v9",
  strictVersion: true
}
```

该侧失败时必须返回失败，不得执行 V8.3 后继续标记为 V9。

#### 未知版本

显式传入未知版本必须直接拒绝。

只有省略版本时，才允许读取生产活动版本。

正式原则：

> 生产允许保命式回退，实验禁止身份伪装式回退。

### 18.5 V9 虫洞必须服从节点出流预算

V9 虫洞只能调制归一化前的原始边强度：

\[
g_{ij}
=
g_{\text{base}}(f_{ij})
\cdot
g_{\text{wormhole}}(B_{ij})
\]

随后与普通边统一归一化：

\[
P_{ij}
=
\frac{g_{ij}}
{\sum_k g_{ik}+\epsilon}
\]

必须满足：

\[
\sum_j P_{ij}\leq1
\]

加入边衰减后仍必须满足：

\[
\sum_j\delta_{ij}P_{ij}\leq1
\]

虫洞可以保留动量豁免，但不得：

- 在归一化后额外增加边能量；
- 绕过节点总出流预算；
- 使用独立于该行预算的注入电流；
- 将低阻通道变成能量生成器。

诊断时必须区分：

- 当前跳在途能量；
- 历次累计场响应。

累计场随有限跳数增长不等同于单跳动力学爆炸。稳定性判断应观察相邻跳质量比和单节点最大放大率。

### 18.6 本轮明确不实施的扩展

为保持工程节制，本轮不实施：

1. 独立的 V8.3 legacy residual 双计算支路；
2. 自动化、机械化的无差异停工阈值；
3. 种子锚质量评分及其额外测量层。

当前兼容策略保持为：

- 旧 `residual_energy` 和已有兼容增益继续服务 V8.3 回退；
- V9 使用单位化 raw residual ratio 与固定增益；
- 不为理论上的完全复刻增加第二套残差预计算复杂度。

当前 A/B 输入由人或 Agent 主动构造，最终裁决继续依据真实寻址结果与主观价值，不额外建设种子质量或机械停工分类器。

### 18.7 修订后的优先级

实施顺序调整为：

1. 原子不可变 `ArtifactBundle`；
2. V8.3/V9 双矩阵与受预算约束的 V9 核；
3. 请求级严格版本和可审计生产回退；
4. LightMemo 模式 A 固定对称超集测核；
5. LightMemo 模式 B 端到端独立寻址测产品；
6. 根据实际结果决定是否继续增加解释层。

任何不能直接保护结果真实性、A/B 公平性或生产安全性的测量机制，不进入本轮核心施工范围。
