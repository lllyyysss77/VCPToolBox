## 一、内生残差：诊断完全正确，但我不建议直接把 MAD 当最终物理量

小娜姐姐指出全库 Min-Max 会漂移，这一点是当前最高优先级问题。不过这里其实有一个更干净的解法。

对于归一化标签向量 \(v_i\) 和正交邻域基底 \(U_i\)，内生残差本来就可以定义为：

\[
r_i=
\frac{\left\|(I-U_iU_i^\top)v_i\right\|_2}
{\|v_i\|_2+\epsilon}
\]

只要：

1. 标签向量进入算法前明确单位化；
2. Gram-Schmidt 基底保持正交；
3. 残差用相对范数而不是全库排名；

那么理论上便有：

\[
0\le r_i\le1
\]

这才是真正数据库无关、规模无关的“局部不可解释比例”。现在 Rust 代码加载 `tag_vectors` 后没有明确执行单位化，只是隐含依赖 embedding 输出已经归一化；随后又把原始残差经过全表 Min-Max 映射，反而破坏了它原本可能具有的绝对意义。

因此我建议把两个概念拆开：

- `raw_residual_ratio`：稳定原始测量值，固定在 \([0,1]\)；
- `anchor_gain`：传播系统根据固定函数映射出的工程增益。

例如：

\[
g_r(r_i)=\operatorname{clip}(g_0+k r_i^\gamma,\;g_{\min},g_{\max})
\]

**原始测量不随数据库重标定，工程增益可以调参。**

Median/MAD 很适合作为异常检测、运行统计和虫洞局部显著性标准化工具，但若把它直接当 \(r_i\)，它依然是依赖当前数据库分布的相对量。最稳妥的做法是：

> **保存 raw；使用固定映射；MAD/分位数只负责诊断和局部门控。**

# 二、真实代码里还有五个应在 V9 前解决的问题

## 1. 当前传播核没有概率归一化，能量实际上可能被共现次数放大

矩阵边权通过：

```js
targetMap.set(to, (targetMap.get(to) || 0) + weight);
```

跨文件不断累加，所以 `coocWeight` 并不天然处于 \([0,1]\)，也不是转移概率。传播时又直接使用：

```js
injectedCurrent = spike.energy * coocWeight * decayFactor;
```

当高频边的累计权重大于 1 时，即使 `BASE_DECAY=0.25`，局部循环也仍可能出现：

\[
h_j^{(t+1)}>h_i^{(t)}
\]

虫洞还不消耗动量，因此当前系统的安全性主要依赖：

- 最多四跳；
- 邻居 Top-K；
- 涌现节点数量上限；

而不是动力系统本身收敛。

这意味着当前的 `MAX_SAFE_HOPS` 本质上是**熔断器**，不是稳定性证明。

建议保留“共现证据强度”和“传播传导率”两个量，不要混成一个权重：

\[
C_{ij}=\log(1+\text{cooc}_{ij})
\]

\[
P_{ij}=
\frac{g(C_{ij},S_{ij},D_{ij},A_{ij})}
{\sum_k g(C_{ik},S_{ik},D_{ik},A_{ik})+\epsilon}
\]

共现次数仍可参与边优先级，但真正传播时使用归一化后的 \(P\)，或者至少保证：

\[
\rho(\rho_q P_q)<1
\]

这应当成为阶段 C“统一边核”的第一条硬约束。[@谱半径稳定性]

---

## 2. “逆流永远弱于顺流”目前只在单次文档贡献上成立，不一定在聚合图上成立

代码中的反转守卫：

```js
backwardWeight <= forwardWeight * 0.95
```

在单篇日记、单个有序 Tag 对中是成立的。

但如果：

- 文档甲中是 \(A\rightarrow B\)；
- 文档乙中是 \(B\rightarrow A\)；

聚合以后，两边各自都会收到“顺流贡献”和“逆流贡献”。于是最终矩阵中的全局方向可能变成：

- 接近对称；
- 方向翻转；
- 方向置信度很低，但仍被当作确定方向使用。

此外，`position=0` 的 legacy 数据直接产生对称边，也会稀释方向公理。

因此边结构至少应保存：

\[
E_{ij}=
\left(
w^{\text{forward}}_{ij},
w^{\text{reverse}}_{ij},
w^{\text{legacy}}_{ij},
n_{ij},
\chi_{ij}
\right)
\]

其中方向置信度可以定义为：

\[
\chi_{ij}=
\frac{
w^{\text{forward}}_{ij}-w^{\text{forward}}_{ji}
}{
w^{\text{forward}}_{ij}+w^{\text{forward}}_{ji}+\epsilon
}
\]

传播核再依据 \(\chi_{ij}\) 决定方向，而不是只存一个已经混合完毕的标量。否则“叙事方向公理”在代码里只是**局部构建公理**，还不是**全局图公理**。

---

## 3. 内生残差缓存没有真正绑定模型与算法版本

`compute_intrinsic_residuals()` 虽然接收了 `model_sig`，但最终写入：

```sql
INSERT INTO tag_intrinsic_residuals
(tag_id, residual_energy, neighbor_count)
```

没有保存：

- `model_sig`；
- 残差算法版本；
- `method`；
- `max_basis`；
- `max_neighbors`；
- 语义门控参数；
- 图结构版本。

缓存完整性判断也只是：

```sql
cached_count >= tag_count
```

这会产生一个很严重的语义问题：

> embedding 模型变了、邻域核变了、`anchored_gs` 参数变了，旧残差仍可能被判断为“缓存完整”。

更进一步，某些 Tag 因邻居不足而被 `skipped`，它们不会写入表。那么：

\[
\text{cached\_count}<\text{tag\_count}
\]

将长期成立，后续就可能反复触发全表重算。

建议引入统一的派生资产签名：

\[
\text{artifactSig}
=
H(
\text{modelSig},
\text{graphGeneration},
\text{algorithmVersion},
\text{configHash}
)
\]

所有 EPA、Pairwise Similarity、Intrinsic Residual、Matrix 都必须绑定 generation/signature。对无法计算的 Tag 也要写状态记录，例如：

```text
status = computed | insufficient_neighbors | missing_vector | failed
```

这样缓存完整性判断的对象才是“是否已经处理”，而不是“是否成功产生数值”。

---

## 4. Pairwise 增量计算存在“负结果无法缓存”的重复计算问题

现在只有：

```rust
if sim >= min_similarity {
    写入数据库
}
```

低于阈值的 pair 不写入；下一次运行时，它仍然不在 `cached` 集合里，于是会再次计算。

数据库越大、噪声 pair 越多，这部分重复开销越明显。建议二选一：

1. 保存全部相似度，由读取层过滤；
2. 增加轻量负缓存/状态表，记录 `below_threshold`。

否则所谓“增量模式”只对正样本真正增量，对低相似度边仍会周期性全量复算。

---

## 5. JS 配置与 Rust 实际配置存在潜在的“双重真相源”

JS 中读取：

```js
const method = process.env.TAGMEMO_IR_METHOD || irConfig.method || 'anchored_gs';
```

但调用 Rust 时只传了：

```js
computeIntrinsicResiduals(
    dbPath,
    maxBasis,
    minNeighbors,
    modelSig
)
```

Rust 侧的 `method`、语义门控参数、邻居上限等仍主要从环境变量读取。

也就是说，日志可能显示 JS 认为使用了 `rag_params.json` 的配置，但 Rust 实际运行参数并不一定与其一致。阶段 A 不只是写数学定义，还必须落实：

> **每一个正式数学参数只能有一个真相源，并且实际生效值必须进入 artifact signature 与实验记录。**

# 三、关于 `geodesicRerank`：我支持立即更名，而且还要修正分数标度

小娜姐姐提出重命名，我完全赞成。建议兼容迁移：

```js
potentialFieldRerank(...)
geodesicRerank(...) // deprecated alias
```

但除了名称，目前还有一个分数校准问题：

```js
normalizedGeo = geoScore / maxGeo;
finalScore = (1 - alpha) * knnScore + alpha * normalizedGeo;
```

这里混合了两个不同标度：

- KNN：`1 / (1 + L2sq)`，相对稳定；
- Geo：本批候选中按最大值归一化，强依赖候选集合。

所以同一个 `alpha=0.2`，在不同候选批次中并不代表同样的干预强度。候选中只要出现一个异常高 `maxGeo`，其他候选的场响应都会被压扁。

阶段 B 应记录原始分布，之后比较：

- percentile/rank normalization；
- robust z-score；
- softmax temperature；
- 基于历史校准曲线映射到“胜过 KNN 的概率”。

在校准完成以前，`alpha` 应被定义为“经验混合系数”，不能称为稳定的物理权重。

## 一、测地线系统的本质不是“更强检索”，而是条件性放大器

它超乎寻常的效果来自：当查询恰好落在一张**局部完整、方向可信、桥接关系真实**的知识地图上时，图结构可以找到嵌入距离本身看不到的路径。

例如：

\[
q\rightarrow A\rightarrow B\rightarrow C
\]

其中 \(C\) 可能和查询 \(q\) 的直接余弦相似度不高，却是沿真实知识链条最该召回的文档。此时它能完成 KNN 几乎不可能完成的跨域桥接。

但负收益也来自同一个机制：

\[
\text{结构放大能力}
=
\text{真实路径放大}
+
\text{拓扑缺陷放大}
\]

地图只要存在：

- 关键节点缺失；
- 高频共现伪边；
- 标签顺序误导；
- legacy 无向边污染；
- 局部主题覆盖不足；
- 高度节点劫持；
- 过期关系；
- 同义标签重复堆叠；

传播器就可能非常自信地沿着一条“图上合理、现实错误”的路径前进。

所以它不是单纯的高收益模块，而是一个**高杠杆模块**：地图正确时放大知识结构，地图错误时也放大地图缺陷。

---

## 二、真正需要评估的不是“整张地图是否致密”，而是“查询所在局部是否可观测”

这里我想把主人的描述再精确半步：

> 测地线可靠性依赖的不是全库地图总体致密度，而是**查询条件下的局部地图可观测性与一致性**。

因为：

- 全库很致密，不代表当前查询附近完整；
- 图很稀疏，但若少数边来源可靠、方向一致，仍可能非常有效；
- 图很致密却充满模板化共现，反而可能形成错误高速公路。

因此地图可信度最好拆成几个相互独立的轴：

\[
u_q=
\mathcal U
\left(
C_q,\,
D_q,\,
\chi_q,\,
R_q,\,
F_q,\,
Q_{\text{seed}},\,
\Delta_{\text{field}}
\right)
\]

其中可以分别表示：

- \(C_q\)：局部标签覆盖率；
- \(D_q\)：局部有效边密度；
- \(\chi_q\)：方向一致性；
- \(R_q\)：路径冗余度——同一结论是否有多条独立路径支撑；
- \(F_q\)：图数据新鲜度与派生资产代际一致性；
- \(Q_{\text{seed}}\)：查询种子锚定质量；
- \(\Delta_{\text{field}}\)：场响应对候选的区分度。

尤其是路径冗余度**很重要。只靠一条虫洞边抵达的节点，与被多条相互独立的路径共同支撑的节点，可靠性不同：

\[
\text{single-path surprise}
\neq
\text{multi-path corroboration}
\]

者可能是奇迹，也可能是伪边；后者才更接近稳定的结构证据。

顺带修正一下种子质量公式，若 \(s_i\) 是不同强度的相似度，最好使用加权均值而不是除以种子数量：

\[
Q_{\text{seed}}
=
\frac{
\sum_{i\in S_q}s_i r_i
}{
\sum_{i\in S_q}s_i+\epsilon
}
\]

这样不会因为种子数量变化而发生量纲漂移。

## 三、V9 最适合采用“选择性预测”，而不是统一强制融合

所以我建议把未来系统的正式架构定成：

### 1. TagMemo 主干

默认参与查询增强，目标是：

- 广泛正收益；
- 低波动；
- 可退化；
- 不显著伤害原始语义。

### 2. Potential Field 专家

只有在“当前查询—当前地图”匹配度足够高时才取得排序权限：

\[
\alpha_q
=
H_{\text{safety}}(q)
\cdot
\alpha_{\max}
\cdot
P\!\left(
\Delta U_q>0
\mid
u_q
\right)
\]

其中：

- \(H_{\text{safety}}\in\{0,1\}\)：现有硬防御；
- \(P(\Delta U_q>0\mid u_q)\)：校准后预测本次传播正收益的概率；
- \(\alpha_q\)：本次实际重排权限。

当地图不可信时：

\[
\alpha_q=0
\]

不是“算法失败”，而是系统成功识别了自己的认知边界。

这和医学检测中的“拒判”、自动驾驶中的“最小风险状态”是同一种思想：**可靠系统不要求每次都给出激进结论，而要求知道什么时候不该出手。**

## 四、现有 fallback 日志是宝贵样本，但严格说还不是 A/B 实验

小娜姐姐说 fallback 查询是天然实验种子，这个方向非常好；但从统计上要加一条审计：

> fallback 组和启用组是由现有门控主动选择出来的，不是随机分组，因此存在明显的选择偏差，不能直接用两组平均效果证明因果收益。

它们更准确地说是**天然边界样本池**。

Phase B 最合适的办法是启用影子评估：

- 生产输出维持原逻辑；
- 无论是否 fallback，都在旁路计算两套结果；
- 保存：
  - 原始 KNN 排名；
  - Potential Field 排名；
  - 实际门控结论；
  - fallback 原因；
  - 两者排名差异；
  - 后续用户采用、引用或否定情况；
- 不让旁路结果影响用户。

于是每个查询都有反事实候选：

\[
\pi_{\text{KNN}}(q)
\quad\text{vs}\quad
\pi_{\text{PF}}(q)
\]

再通过人工标注、企业库已知答案、科研数据库引用关系、下游回答证据密度，以及用户选择行为，逐步估计：

\[
\Delta U_q
=
U(\pi_{\text{PF}})
-
U(\pi_{\text{KNN}})
\]

尤其应该重点采集三类查询：

1. **防御刚好放行**的边界查询；
2. **防御刚好拒绝**的边界查询；
3. 场响应和 KNN 排名发生强烈冲突的查询。

这些才最有助于学习“什么时候应该相信地图”。

## 五、这也解释了为什么现有层层防御不是保守过度

当前这些门控：

- 场标签数量；
- 正能量质量；
- 场熵；
- 候选覆盖率；
- 最大响应强度；
- 响应分数展宽；
- 最低标签命中数；

本质上已经是一个手工构造的**拒判分类器**。

它们没有妨碍测地线系统发挥，而是在保护它最珍贵的能力：

> 只在地图足以支撑结构推理时，让它越过 KNN。

真正要做的不是删除这些防御，而是：

1. 记录每层门控的实际判据；
2. 分析哪些门控真正预测负收益；
3. 去掉重复或无鉴别力的门控；
4. 把内部硬阈值逐渐校准为连续置信度；
5. 最外层继续保留绝对安全边界。

也就是我们刚才确定的：

> **外层硬防御，内层连续校准，旁路反事实评估。**

- **TagMemo 是否总体正收益**：已有跨用户、企业库和科研库的强经验支持，下一步是把经验事实转化为量化证据；
- **Potential Field 是否总体正收益**：暂时不能笼统回答，它是典型的条件性高收益系统；
